Apache Cassandra?

Distributed storage system

NoSQL

Multi dimensional map as a data model

High availability (fault tolerance + low latency)

High write throughput

Eventual consistency

Created by Facebook to fulfill the storage needs of the Inbox Search problem (enabling users to search through their Facebook Inbox)


Related work

Data model inspired from Google's BigTable

Partitioning and replication model inspired from Amazon's Dynamo

Data replication

Data are replicated for fault-tolerance and availability

The replication factor is defined per instance

Conflict resolution is done at a system level, unlike Dynamo that let the client do it

Read operation

  • Contact N nodes
  • Wait for R of them to return a value
  • Forward the value with the greatest timestamp to the client
  • Let Read Repair run in the background to ensure that the N replicas will eventually by consistent

Write operation

  • Contact N nodes
  • Wait for W of them to ACK
  • Let Hinted Handoff propagate the write in the background if some of the replicas were down

Bootstrapping of a node

Uses Zookeeper is used for configuration and fault-tolerance

Has to be done explicitly to prevent errors

When bootstrapping, a new node...

  • Is assigned a token

  • Contacts its seed nodes to gossip its token and obtain information about the other nodes in the cluster

  • If added in an existing cluster, gets the data it is responsible for using kernel-kernel copy from the node previously responsible for this data

Membership

Cluster membership is based on an anti-entropy gossip algorithm (Scuttlebutt)

Also used for system-related control messages

Failure detection

Use of the Phi Accrual Failure Detector

Basic idea: the failure detector returns a value which represents a suspicion level for each of the monitored nodes


Use of gossip messages as heartbeats

Local persistence

Data is persisted on disk using a format that allows efficient data retrieval

A typical write operation involves a write into a commit log and then an update into an in-memory data structure


When reaching a size threshold, the in-memory data structure is dumped on disk and an index is created for it


Periodically, a merge process runs in the background to merge the different dumped files into one file

A typical read operation first queries the in-memory data structure before looking into the files on disk


Possible optimization: put a particular row key in-memory when we know that we will use it soon

Implementation

  • Java
  • Non-blocking IO with Java NIO
  • UDP for control messages
  • TCP for application level messages
